1. Descrição do Sistema

O ClearIT Copilot é um assistente de inteligência artificial desenvolvido para auxiliar analistas de suporte técnico Nível 1 (L1) no diagnóstico e resolução de incidentes. O sistema busca por significado (não por palavra-chave), conectando problemas atuais com soluções já encontradas anteriormente — mesmo quando descritos com termos completamente diferentes.

O sistema é consultivo: ele sugere diagnósticos e ações, mas a decisão final é sempre do analista humano. Nenhuma ação é executada automaticamente.

Como o sistema resolve o problema

  1. O analista descreve o problema com suas próprias palavras
  2. O texto passa por mascaramento de dados sensíveis (LGPD)
  3. O sistema converte o texto em representação vetorial (embedding)
  4. Busca por similaridade em 3 fontes: tickets históricos, bases de conhecimento e API do FreshService
  5. O modelo Gemini gera um diagnóstico estruturado com base nas fontes encontradas
  6. O resultado inclui: diagnóstico, passos de ação, fonte citada e nível de confiança

2. Como Usar

2.1 POC — Demonstração Local

Pré-requisitos

  • Node.js 18 ou superior instalado
  • (Opcional) Chave de API do Gemini — Google AI Studio

Instalação

git clone https://github.com/beandy-dev/sherlock-clearit-copilot.git
cd sherlock-clearit-copilot
npm install
cp .env.example .env

Configuração

Edite o arquivo .env (opcional — sem ele o sistema funciona em modo demo):

GEMINI_API_KEY=AIzaSy...sua_chave_aqui

Execução

npm start

Acesse http://localhost:3000 no navegador.

Passo a passo de uso

  1. Na lista de tickets à esquerda, clique em um chamado para ver seus detalhes
  2. Na sidebar direita (ClearIT Copilot), o campo já vem pré-preenchido com a descrição do ticket
  3. Clique "Analisar" para solicitar o diagnóstico
  4. Aguarde a resposta (2-15 segundos, dependendo do modo):
    • Modo Demo (sem API key): resposta instantânea com dados simulados
    • Modo Live (com API key): busca semântica real + geração com Gemini
  5. O resultado exibe: diagnóstico, ações recomendadas, fonte e nível de confiança
  6. Após o resultado, você pode:
    • Dar feedback (Útil / Parcial / Não ajudou)
    • Regerar a resposta
    • Buscar na web (pesquisa documentação de fabricantes)
    • Chat — fazer perguntas adicionais mantendo o contexto
Dica: No modo demo, os 6 tickets da lista possuem respostas pré-definidas de alta qualidade. Use-os para demonstrações sem risco de falha.

2.2 Widget FreshService (MVP)

Pré-requisitos para instalação

  • Node.js 24.11 (via nvm install 24.11)
  • FDK 10 (npm install https://cdn.freshdev.io/fdk/latest-v24.tgz -g)
  • Conta de administrador no FreshService

Empacotamento

git clone https://github.com/beandy-dev/clearit-copilot-freshservice-api.git
cd clearit-copilot-freshservice-api
nvm use 24.11
fdk validate --app-dir .
fdk pack --app-dir . --skip-coverage

Instalação no FreshService

  1. Acesse Admin → Apps → Custom Apps
  2. Clique New App → Custom App
  3. Upload o arquivo .zip gerado na pasta dist/
  4. Preencha o nome e salve
  5. Na tela de instalação, configure a URL do Backend (ex: https://seu-backend.onrender.com)
  6. Instale

Uso

  1. Abra qualquer ticket no FreshService
  2. O widget aparece na sidebar direita com a descrição já pré-preenchida
  3. Clique "Buscar Diagnóstico"
  4. Mesmas opções da POC: feedback, regerar, buscar na web, chat
Atenção: O FDK 10 exige Node.js exatamente na versão 24.11.x. Versões superiores (24.15+) ou inferiores (20, 18) não funcionam para empacotar. Porém para rodar o backend, qualquer Node 18+ funciona.

Durante a instalação do widget no FreshService, o administrador configura:

ParâmetroDescriçãoExemplo

Servidor temporário: O backend em clearit-copilot.onrender.com está disponível para testes e avaliação enquanto o programa Pulse Mais 2026 estiver ativo. O widget funciona sem configuração adicional enquanto esse servidor estiver no ar. Após desativação, será necessário fazer deploy próprio seguindo as instruções do repositório clearit-copilot-backend.
Importante: O backend deve ser acessível via HTTPS. O FreshService bloqueia requisições HTTP (Mixed Content). Serviços como Render já fornecem HTTPS automaticamente.

Obter a chave do Gemini

  1. Acesse Google AI Studio
  2. Clique "Create API Key"
  3. Copie a chave (começa com AIzaSy)
  4. Cole no .env ou nas variáveis de ambiente do Render

5. Possíveis Problemas e Soluções

SintomaCausa provávelSolução
Loading por mais de 30 segundosGemini demorou para responder (sobrecarga ou rate limit)O sistema exibe mensagem de timeout automaticamente. Clique "Nova consulta" e tente novamente após alguns segundos.
Resposta genérica ou de baixa qualidadeSistema operando sem API key (modo fallback)Configure GEMINI_API_KEY válida no .env e reinicie o servidor.
"Não foi possível conectar ao backend"Backend offline, URL errada ou cold start do RenderAguarde ~30 segundos (cold start) e tente novamente. Verifique se a URL do backend está correta.
Widget não aparece no FreshServiceApp não instalado ou desabilitadoAdmin → Apps → verificar se o ClearIT Copilot está ativo.
Feedback não confirma envioCold start do Render (~30s primeira request)O sistema tenta 2x automaticamente. Se falhar, tente novamente após o backend acordar.
Busca web sem resultadosProblema muito específico sem referências no treinamento do GeminiNormal em cenários de nicho. Use o chat para pedir informações mais genéricas sobre o tema.
Confiança muito baixa (<70%)Problema não existe na base interna (40 tickets + 13 KBs)Esperado. O sistema avisa automaticamente e busca na web. Use "Buscar na web" para complementar.
Erro EADDRINUSE ao iniciarPorta 3000 já está em usoExecute kill $(lsof -t -i:3000) e inicie novamente.
Primeira consulta demora 10-20sCache de embeddings sendo gerado pela primeira vezNormal. O sistema gera embeddings para 40 tickets + 13 KBs. Consultas seguintes são rápidas (~2s).
"Platform version below 3.0" ao subir widgetManifesto antigoUsar platform-version: "3.0" sem campo product no manifest.json.
FDK não aceita Node 20FDK 10 exige Node 24.11nvm install 24.11 && nvm use 24.11 antes de empacotar.

6. Recomendações de Uso

Para obter melhores resultados

  • Descreva com detalhes — "Backup falhou" é vago. "Commvault job de backup deu timeout no storage NetApp às 2h, servidor PROD-DB-01" gera diagnóstico preciso.
  • Inclua contexto técnico — mencione o produto (Nutanix, Commvault, Cisco), o ambiente (produção/homologação) e quando começou.
  • Verifique a fonte — toda sugestão indica de onde veio (Ticket #X, KB #Y). Valide antes de aplicar.
  • Use o chat para comandos — após o diagnóstico, pergunte "qual comando específico?" ou "como verifico isso?".
  • Dê feedback sempre — isso melhora o sistema. "Parcial" com edição é mais valioso que "Útil" sem contexto.

O que NÃO fazer

  • Não aplique sugestões sem validar — o sistema pode errar. Confiança <70% merece atenção redobrada.
  • Não cole dados sensíveis reais esperando que apareçam na resposta — eles serão mascarados. O diagnóstico funcionará normalmente, mas os dados reais não serão visíveis na saída.
  • Não use para problemas fora do domínio — a base foi construída para infraestrutura (Nutanix, Commvault, VPN, AD, IIS). Problemas de software aplicativo terão confiança baixa.
Princípio do sistema: Uma sugestão errada aplicada sem validação é pior que nenhuma sugestão. Por isso o sistema sempre mostra a fonte e o nível de confiança — para que o analista tome a decisão informada.

8. Próximos Passos e Evolução

Curto prazo (pós-entrega)

  • Autenticação — JWT ou API key por instalação. Proteger a API.
  • Testes automatizados — Unit + integration tests para garantir estabilidade.
  • Logging estruturado — Tempo por etapa (mascaramento, embedding, busca, geração). Alertas para erros.
  • Pipeline de curadoria de feedback — Antes de re-ingerir: verificar duplicidade (>85%), temporalidade (>24h), volume (2+ analistas), revisão humana.

Médio prazo

  • Cache inteligente — Mesma consulta em 30 min → retorna cache sem chamar Gemini.
  • Dashboard de métricas — Taxa de aceitação por categoria, gaps de KB, temas recorrentes.
  • Google Custom Search API — Links reais verificados + informações atualizadas em tempo real. Atualmente o Gemini usa conhecimento do treinamento (pode estar desatualizado). Com busca real, o sistema traria documentação vigente dos fabricantes com link direto para a fonte.
  • Webhook proativo — Alerta do Zabbix/Datadog → pré-diagnóstico automático → ticket nasce com sugestão.

Longo prazo

  • Fine-tuning — Modelo especializado no domínio ClearIT após 500+ interações curadas.
  • Multi-tenant (SaaS) — Base isolada por cliente, billing por uso.
  • Integração Slack / Teams — Bot com diagnóstico direto no canal de suporte.
  • Análise de tendências — Identificar picos de problemas similares e alertar o gestor proativamente.
Sugestão da banca avaliadora (11/07/2026): Implementar regulação sobre o que é aceito como entrada para melhoria do banco de dados. Respostas aprovadas não devem ser re-ingeridas automaticamente — precisam passar por validação de duplicidade, análise temporal e revisão por analista sênior antes de enriquecer a base vetorial.
